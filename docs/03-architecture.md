# Architecture

The first release is a Go modular monolith using chi, pgx, explicit SQL, PostgreSQL, Redis Streams, and an S3-compatible object store. Transactional writes place domain events in the PostgreSQL outbox. Workers deliver them to Redis Streams and deduplicate inbox records.

Web applications live in a pnpm workspace. Intelligence and crawler workers are Python processes behind typed adapters. Deployment starts with Docker Compose; later Kafka, Temporal, OpenFGA, OpenMeter, and analytical stores remain replaceable adapters.

The transaction boundary starts from an accepted immutable QuoteVersion. Order creation copies every ProductVersion, SKUVersion, Workflow, Pricing, and Routing identifier into OrderItem facts. Moving an order to provisioning creates Subscription, Entitlement, ExecutionOrder, and DeliveryProject records in the same tenant-scoped transaction. Usage, ProviderCost, and CustomerCharge remain separate operational facts. Explicit finance commands convert eligible facts into account-locked, append-only, balanced ledger transactions; wallets, holds, refunds, payables, commissions, settlements, and reconciliation remain separately auditable records.

The growth plane is a separate PostgreSQL/RLS boundary. MarketSegment and ICPDefinition describe target criteria; Lead and Contact remain distinct from Customer; Proof is Schema/workflow driven; Campaign activation requires approval for its exact version; suppression is checked before quota reservation; and outreach is persisted only as `planned` or `blocked`. The browser API cannot advance outreach into delivery while `growth.outbound_delivery` is disabled.

Deal is the growth pipeline aggregate, not an Order. A canonical Deal creates the existing transaction-domain Quote through `quotes.growth_deal_id`; the historical text `deal_id` remains available for external or pre-phase-F quotes. Deal proposal and win transitions require corresponding draft and accepted canonical Quote states.

The channel and internal-marketplace boundary is one tenant-scoped PostgreSQL module. Reseller ownership references Growth Lead and the existing customer identifier. CommissionLock references the canonical finance Commission; Supplier owns existing Provider records, while supplier views project canonical ProviderPayable and Settlement facts. Developer/Publisher Listings are versioned separately from Product and workflow facts. Listing publication requires persisted review, sandbox, and quality gates; takedown changes Listing state without deleting its versions, reviews, incidents, disputes, or audit history.

Phase H closes the learning loop without introducing another commercial fact model. Source and Signal are persisted and fingerprint-deduplicated before an atomic promotion creates Opportunity evidence and lineage. A queued ExecutionOrder starts one persisted WorkflowRun; side-effect nodes receive bounded leases tied to a Provider-bound Adapter identity. Browser sessions cannot supply external execution results. Signed Adapter receipts verify HMAC, timestamp, nonce, persistent Inbox deduplication, endpoint identity, and active lease before execution, step, audit, and Outbox facts commit together.

OutcomeFeedback is accepted only when the Order originates from the Opportunity's BusinessBlueprint, every execution is settled, charges are posted, payables and commissions are settled, and a matched ReconciliationRun exists. Revenue, provider cost, commission, and gross margin are derived from canonical integer facts. The latest projection is written to Opportunity for reads while every feedback version remains append-only.
